# 月舟(tsuki-bune) 仕様メモ

## コンセプト

月舟は、MarkdownとWordの中間のような日記アプリです。

MarkdownはAIやGitには読みやすい一方、文字色、文字サイズ、下線、ハイライト、画像コピペなどの表現が弱いです。Word/docxは表現力が高い一方、本文、スタイル、画像、メタデータが複雑にパッケージ化され、AIやGitから扱いにくくブラックボックス感があります。

月舟では保存形式をHTMLにし、Wordのようにリッチに編集できるが、AIにも人間にも読みやすい日記データを目指します。

月舟は自分でも使うアプリですが、自分専用ではなく、老若男女誰でも使える簡単なアプリを目指します。

エージェントAIの時代には、自分の情報、日記、生活ログ、考えたことを「人生レポジトリ」として蓄積しておく価値が高くなります。AIモデルは賢くなっていても、その人の背景情報を知らないために、気の利く答えを出せないことがあります。人間にとっては簡単に見える依頼でも、実際には本人の生活、性格、過去の経験、予定、好み、人間関係などの背景情報が必要な場合があります。

人生レポジトリがあれば、エージェントAIはそれを参照し、背景を理解したうえで返答や作業ができるようになります。大きなコンテキストウィンドウを持つAIモデルの力を生かすうえでも、AIが読める形で日々の情報を蓄積することは重要です。

一方で、今の人生レポジトリ運用はエンジニアに寄りすぎています。GitHub、VS Code、Markdownを使いこなす必要があり、普通の人にはハードルが高いです。MarkdownはAIやGitには扱いやすい反面、記法を覚える必要があり、文字色、スクリーンショット、ハイライトなどの直感的な表現は弱いです。Wordは直感的ですが、保存形式が複雑でAIやGitには扱いにくいです。

月舟は、この間を埋めるためのアプリです。HTMLを保存形式にし、Wordのように簡単に編集でき、AIにも人間にも読める人生レポジトリを作れるようにします。

授業のメモ、アニメの感想、今日起こったこと、明日やりたいこと、長文メッセージの下書きなど、日常のあらゆる文章を書く場所として使えることを目指します。

基本はローカル管理です。ユーザー自身のパソコン内にデータを置き、パソコンを守れば日記も守れる、という安心感を大事にします。

## 名前

開発段階の名前は「月舟(tsuki-bune)」です。

月はAI、地球は人間の生活を表します。AIは知的でも、人が何をしているのか、どういう人なのか、どんな人生を歩んできたのか、日々何をしているのかを知らないことがあります。人とAIをつなぐテキストを、このアプリでたくさん作ってほしいという願いから名付けています。

## 初期技術判断

プロトタイプでは静的HTML/CSS/JavaScriptを採用します。

理由:

- 依存関係が少なく、すぐ動く
- Docker配信が単純
- HTML保存という中核価値を最短で試せる
- 将来のエディタ差し替えがしやすい

今後の候補:

- 本格エディタ: TipTap/ProseMirror
- デスクトップアプリ化: Tauri
- ローカルファイル保存の改善: File System Access API
- 同期や複数端末対応: 後続検討

## 将来像

月舟はWebブラウザで動くアプリを基本にしたいです。

ただし、単にブラウザ内だけで完結するメモアプリではなく、ローカルファイルやGitHub上の「life-repo」のような人生管理リポジトリ内のHTML日記を編集できる体験を目指します。

単独HTMLファイルの読み込みはフォールバックとして実装します。将来的には、以下のようなフォルダ単位の体験を本命にします。

- ブラウザでアプリを開く
- ユーザーがローカルの日記フォルダを選ぶ
- アプリがフォルダ内のHTML日記を一覧表示する
- 日記を編集して同じファイルへ保存する
- そのフォルダをGit/GitHubで管理する

技術候補:

- File System Access API: Chromium系ブラウザでローカルフォルダを扱う
- GitHub API: GitHub上のリポジトリを直接読み書きする
- Tauri/Electron: ブラウザ体験を保ちつつローカルファイル操作を強くする

## 体験の原則

月舟は、パソコン初心者でも使いやすく、入り組んだ操作を避けることを重視します。

原則:

- 主要操作は画面を見れば分かるようにする
- ショートカットは便利機能として用意し、必須操作にはしない
- 機能を増やす場合も、既存の軽快さと明瞭さを優先する
- 日記本文と保存HTMLの読みやすさを壊さない
- レアアイコンやアニメーションなどの遊びは、軽量性を損なわない範囲のサブ要素として扱う
- 説明を読まなくても、開いたらすぐ書けることを優先する
- カスタマイズ項目を増やしすぎず、使っているうちに自然に覚えられる操作感を目指す

## デザインの方向性

初期段階では、デザインの作り込みより機能の整理を優先します。

方向性:

- 静かな日記帳のように、文章を書くことを邪魔しない
- WordやGoogle Docsのように、多くの人がなじみやすい文書編集体験に寄せる
- 業務ツールや管理画面のように堅くしすぎない
- メモアプリとして自然に使える軽さを保つ
- 月舟らしい遊び心は少し入れてよいが、本文編集や起動の軽さを邪魔しない
- 左上の月舟アイコンは、将来的に低確率で短いレア演出を出す余地を残す。演出は無音、起動時に一度だけ、再生後は通常表示に戻ることを基本にする
- 開発会話では左側の操作領域を「サイドバー」、ユーザー向け説明では「左パネル」と呼ぶ
- 本文を書く白い紙面風の領域は当面維持し、フォーカス時の枠色は静かな青緑系を基本にする
- 本格的なデザイン調整は、主要機能がまとまってから行う

## データの約束

月舟は、ユーザーの文章と画像をユーザー自身のものとして扱います。

基本方針:

- 基本はローカルの日記フォルダに保存する
- 最初はユーザーが「自分の日記フォルダはここ」と選べる体験を本命にする
- PC初心者向けには、将来的に保存先を迷わせない導線も検討する
- クラウド保存やGitHub連携は後続検討であり、月舟の中核ではない
- アカウントなしで基本機能を使えるようにする
- 日記内容を月舟側が勝手に外部送信しない
- 日記内容を月舟の運営側サーバーへ保存しない
- 日記フォルダ全体を勝手に読み込んで外部送信することはしない
- AI連携は将来機能であり、最初の段階ではHTML編集ソフトとして成立させる
- AI連携を入れる場合も、ユーザーが明示的に選んだ範囲だけを対象にする
- 現時点では、月舟内にAI機能を入れるより、AIが読みやすいHTMLを残すことを優先する
- ユーザーの文章はユーザーのものだと明確にする
- 月舟を使わなくなっても、HTMLと `assets/` の画像は残る
- ブラウザ内の一時的な保持は保険として使うが、HTMLファイル保存と混同させる表示はしない

保存形式:

- 保存形式はHTMLを本命にする
- 月舟なしでも、ブラウザやエディタで最低限読める状態を保つ
- 月舟専用の属性や構造が多少入ることは許容するが、本文や画像がブラックボックス化しないようにする
- HTMLはAIが読みやすい素直な構造を基本にする
- 画像はHTMLに巨大なData URLとして埋め込まず、`assets/` で別管理する

公開時の約束:

- 広告や過度なトラッキングより、寄付や支援リンクの方が月舟らしい
- アクセス解析を入れる場合も最小限にし、日記内容には触れない
- 「どんなことでも恥ずかしがらずに書ける」安心感を大事にする
- 日記内容を運営側が見られない設計を優先する

## やらないこと

月舟は「開けた瞬間に使える」「開くのも楽、開いてからも楽」なアプリを目指します。

避けること:

- アカウント必須にする
- 日記内容を運営側サーバーに保存する
- GitHub連携やAI連携がないと使えない状態にする
- Notionのように多機能で複雑な画面にする
- 設定項目だらけにする
- 重いアプリにする
- 遊び心やかわいさが本文編集の邪魔をする
- ブラウザでHTMLを開いても中身が読めないほど独自形式に寄せる

クラウド同期について:

- 月舟が公式に重いクラウド同期を中核機能として抱えることはしない
- ユーザーが自分でクラウド同期フォルダを日記フォルダとして選ぶ運用は妨げない
- その場合も、月舟から見ればローカルフォルダを編集しているだけという扱いにする

軽量性の目標:

- 「こんなに軽いのか」と驚かれるくらいの軽さを目指す
- 起動、保存、一覧表示、本文編集のすべてでサクサク感を優先する
- メモリを食わず、開きっぱなしにできるアプリを目指す

## 編集ショートカット

ショートカットは、Wordや一般的なエディタに近い自然な体験を目指します。

候補:

- `Ctrl+B`: 太字
- `Ctrl+U`: 下線
- `Ctrl+S` / `Cmd+S`: 現在の日記HTMLを保存、または上書き保存
- 取り消し線は太字、下線と同じくツールバーから使えるようにする
- 見出し/本文は、本文に戻す操作を含めて、見た目で分かる軽いメニューにする
- 文字色は黒、赤、青、灰色の小さいパレットを基本にする
- 蛍光ペンは黄色、緑色の小さいパレットを基本にする
- 自由に選べるカラーピッカーは初期段階では置かず、基本パレットに絞る
- 文字サイズは「小(80%)、標準(100%)、中(150%)、大(200%)」を、実際の大きさが分かる軽いメニューで選ぶ
- 日記フォルダ未選択で保存がダウンロードになる場合は、事前に確認する
- ユーザーが編集したあとは、ポップアップではなく「未保存の変更があります」と表示して保存を促す

注意:

- ブラウザ標準ショートカットと衝突しすぎない
- パソコン初心者にも分かるよう、必要ならツールチップやヘルプに表示する
- ショートカットが使えなくてもボタン操作で同じことができる

## 日記一覧と移動

日記フォルダの一覧は、最近の日記から見つけやすい表示を基本にします。

日記ファイルの単位:

- `YYYY-MM-DD-タイトル.html` のように、日付とタイトルを含むHTMLファイルで管理する
- 同じ日に複数の日記、メモ、感想、下書きを作れる
- 日記だけでなく、授業メモ、アニメ感想、長文メッセージの下書きなども同じ形式で扱う
- フォルダ構成は、まず1フォルダ直下にHTMLを並べるシンプルな形にする
- 年/月フォルダ分けは、必要になるまで導入しない

候補:

- 日付が新しい日記を上に表示する
- 必要に応じて編集日時が新しい順も検討する
- 日記が増えたら検索や絞り込みを追加する
- 複数HTMLを行き来する体験は、ブラウザの複数タブ、アプリ内タブ、一覧の操作性改善のどれが軽くて分かりやすいか検討する

現在の方針:

- ファイル名先頭の `YYYY-MM-DD` を見て、日付が新しい順に並べる
- 同じ日付の日記はファイル名順で並べる
- 日付が読み取れないHTMLは一覧の下側に回す
- 検索は小さい検索窓で、ファイル名だけを即時絞り込みする
- 本文検索は重くなりやすいため、初期段階では行わない

現時点では、アプリ内に複雑なタブ機能を入れるより、まず日記一覧と検索を強くする方針が有力です。

## 公開とマネタイズ

将来的にはWebサービスとして公開する可能性があります。

方針:

- 基本価値は無料で試しやすくする
- お小遣い程度の支援導線として、コーヒー代をおごるような寄付・支援リンクを検討する
- マネタイズ要素は編集体験の邪魔にしない
- 軽さ、明瞭さ、保存データの扱いやすさを優先する

## 軽量性

月舟は、どんなブラウザでも基本機能が動き、とても軽量で、常時タブで開いておきたくなるアプリを目指します。

方針:

- 依存ライブラリを増やしすぎない
- 起動時に大きな処理をしない
- 日記ファイルは必要になったときだけ読む
- ブラウザ非対応機能はフォールバックを用意する
- ローカルフォルダ直接保存はFile System Access API対応ブラウザの拡張機能として扱う

## 画像

日記フォルダを選択している場合、貼り付け画像はHTML内にData URLとして埋め込まず、日記フォルダ直下の `assets/` に保存します。

HTMLには相対パスだけを残します。

```html
<img src="./assets/2026-05-11-test-001.png" alt="">
```

方針:

- 画像ファイル名は日記HTMLのファイル名ベースにする
- 画像形式は貼り付け元の形式をなるべく維持する
- タイトルや日付の変更で日記HTMLのファイル名が変わる場合、HTMLが参照している画像だけリネーム相当で追従する
- 保存時に本文から参照されていない同一日記ベースの画像は `assets/` から削除する
- フォルダ未選択時や非対応ブラウザでは、従来どおりData URL埋め込みにフォールバックする
