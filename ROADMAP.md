# ROADMAP

月舟(tsuki-bune)のやりたいこと、優先順位、将来案を整理するファイルです。

迷ったときは「サクサク気軽に、使いやすい」「軽い」「明瞭」「初心者でも迷いにくい」を優先します。

月舟の大きな目的は、エンジニアでなくても人生レポジトリを作れるようにすることです。

## 今すぐやる

- 初心者が迷いにくい日記フォルダ選択の導線を考える

## 完了

- `Ctrl+S` / `Cmd+S` で現在の日記HTMLを保存、または上書き保存できるようにする
- 日記フォルダ一覧を日付が新しい順に表示する
- 編集後に「未保存の変更があります」と表示する
- 日記一覧をファイル名で検索、絞り込みできるようにする
- 文字色と蛍光ペンの基本パレットを追加する
- 文字サイズを「小(80%)、標準(100%)、中(150%)、大(200%)」の見た目で選べる軽いメニューにする
- 編集ツールバーを分かりやすくする
  - 見出し/本文の切り替えを確実に効くようにする
  - 文字サイズメニューのボタン名を「文字サイズ」にする
  - 取り消し線を追加する
  - 「クリア」を「書式クリア」にする
- 本文が長くても書式設定リボンが使いやすいように、ツールバーを上部に固定する
- 選択中に文字色変更が見えなくなる問題を避けるため、本文の選択表示を薄い背景にする

## 近いうちにやる

- 日記フォルダを選んでから編集する流れを自然にする
- 日記一覧の並び替え切り替えを検討する
  - 編集日時が新しい順
  - ファイル名順
- HTML保存形式をさらにきれいにする
- ショートカット一覧をどこかで確認できるようにする
- 日記削除や画像削除を取り消しやすくする方法を検討する

## 検討中

- 複数HTMLの行き来をどうするか
  - ブラウザの複数タブで運用する
  - アプリ内タブを作る
  - 一覧、検索、保存状態表示を強くして単一画面のまま運用する
- 文字色や蛍光ペンのショートカットを将来入れるか
- 画像の軽量化や圧縮を入れるか
- GitHub上のlife-repoを直接扱うか
- Tauriなどでローカルアプリ化するか
- クラウド保存を将来入れるか
- AI連携を将来入れるか
- 機能がまとまった後に、静かな日記帳とGoogle Docs的ななじみやすさを両立するデザインへ整える
- 本文を書く白い紙面風エリア、フォーカス時の枠色、サイドバー/左パネルを含めた全体デザインを相談する

## やらない・慎重に扱う

- アカウント必須にはしない
- 日記内容を勝手に外部送信しない
- 日記内容を運営側サーバーに保存しない
- 日記フォルダ全体を勝手に読み込んで送信しない
- カスタマイズ項目を増やしすぎない
- GitHub連携を必須にしない
- クラウド保存を中核機能にしない
- 初期段階でアプリ内AI機能を入れない
- 最初から年/月フォルダ分けを導入しない
- 初期段階でデザイン作り込みに寄りすぎない
- Notionのような多機能で複雑な画面にしない
- 遊び要素が本文編集の邪魔をしない
- 重いアプリにしない

## 許容する運用

- ユーザーが自分でクラウド同期フォルダを日記フォルダにすることは許容する
- ただし月舟本体はローカルフォルダ編集ソフトとして扱う

## 遊び要素

- 月舟アイコンが低確率でレア表示になる
- 軽いGIFや小さなアニメーションで、アイコンに少し動きを出す
- 起動時に一度だけ無音で再生し、再生後は通常の静止アイコンへ戻す
- 通常、ちょいレア、レア、超レアのように確率差をつける
- 最初はCSSアニメーションで軽く試し、リッチ演出が必要になったらAnimated WebPなどを必要時だけ読み込む

条件:

- 起動や編集が重くならないこと
- 本文編集の邪魔をしないこと
- なくてもアプリの本体価値が成立すること

## 将来の公開

- Webサービスとして公開する可能性がある
- 支援リンクやコーヒー代のような軽いマネタイズを検討する
- マネタイズは編集体験の邪魔にしない

## GitHub Issuesへの移行目安

今はこのファイルで十分です。

GitHub Issuesを使い始める目安:

- タスク数が10個以上になった
- 完了済み、進行中、未着手を分けたくなった
- バグと機能要望を分けたくなった
- コミットやPRとタスクを紐づけたくなった
